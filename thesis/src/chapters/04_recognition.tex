\chapter{Card Recognition Pipeline}\label{chap:rec}

\section{Data Acquisition}
As was previously mentioned in \refsec{sec:scarcity}, one of the primary challenges in training deep learning models for \ac{TCG} identification is \ii{data scarcity}. While the total number of unique cards in circulation is vast (as of 2026, \ac{MTG} alone features approximately 30,000 mechanically unique cards by name, a number that expands to over 100,000 distinct target classes when accounting for reprints, alternate artworks, and special foil treatments in the English language alone~\cite{scryfall}), labeled datasets containing real-world photographs of these cards are virtually non-existent. 

Furthermore, collecting and manually labeling a sufficiently large dataset of physical cards is prohibitively expensive and time-consuming. This constraint serves as a secondary motivation for the hardware proposed in this thesis: once constructed, the robotic sorting mechanism can function as an automated data acquisition pipeline. By utilizing models trained on synthetic data to bootstrap the initial sorting, the robot can mechanically feed, photograph, and confidently pseudo-label physical cards, thereby autonomously generating a large-scale, real-world dataset to fine-tune future iterations of the recognition model.

To address the lack of real-world photographs, this thesis adopts a \ii{synthetic data generation} approach. High-quality reference images, serving as ground truth templates, were retrieved via the Scryfall Bulk Data API~\cite{scryfall}\footnote{Scryfall is a specialized search engine and comprehensive database for \ac{MTG}. Users can query cards using advanced syntax to filter by name, color, set edition, gameplay mechanics and format legality. Due to its maintained catalog, it serves as an essential tool for both players and third-party developers.}. This service provides a comprehensive catalogue of all printed cards in a structured JSON format.

However, relying exclusively on digital templates introduces a secondary challenge, so called \ii{Domain Shift}\footnote{See \refsec{sec:domain_shift}.}: the model must be trained to generalize from pristine digital scans to physical photographs captured under variable real-world conditions. Physical environments inherently introduce uneven lighting, lens distortion, glare and/or lower camera resolutions. To bridge this visual gap, the digital templates are processed through a \ii{synthetic data generation} pipeline (defined in \refsec{sec:synthetic_gen}), enabling the automated creation of training samples that computationally simulate these real-world environmental artifacts.

\subsection{Scraping Pipeline and Indexing}
A custom data collection script, \texttt{data\_scraper.py}, was developed to process metadata and download high-resolution scans. A simple file indexing strategy was also designed, aimed at achieving a balance between machine uniqueness and human readability, a feature essential for manual inspection and debugging.

Instead of relying solely on human-unreadable identifiers (such as \acp{UUID}), a hybrid naming convetion derived from the card's canonical \ac{URI} is proposed. Each file name combines the card edition, collector number, and name with a \ac{UUID}: 
\begin{equation}
    \texttt{filename} = \texttt{edition\_collectorNumber\_name-UUID.png}
\end{equation}
For example, a card from the \ii{Zendikar} set is stored as:
\begin{center}
    \texttt{zen\_21\_kor-outfitter-00006596-1166-4a79-8443-ca9f82e6db4e.png}
\end{center}
This format allows for immediate visual identification of the card's edition and variant without requiring a lookup table, while the \ac{UUID} suffix guarantees collision-free storage.

\subsection{Handling Complex Layouts}
The script also implements logic to handle complex card layouts, such as \acp{DFC} (e.g., transform and modal \acp{DFC})\footnote{\acp{DFC} deviate from normal \ac{MTG} cards by featuring playable frames on both the front and back faces, lacking a traditional card back~\cite{mtg_rules}. Each side thus functions as a separate visual unit.}. These entries, identified by the \texttt{layout} property in the JSON metadata, contain multiple faces within a single object. The pipeline detects these instances and automatically splits them into separate template images, appending \texttt{\_front} or \texttt{\_back} suffixes to their identifiers. This ensures the network learns to independently classify either face of the physical card without conflating the features of both sides.

When a neural network is trained exclusively on clean digital templates but deployed to recognize physical cards captured by a camera, it typically experiences a severe degradation in predictive accuracy; a phenomenon documented as \ii{domain shift}~\cite{quinonero2009dataset}. The challenge of adapting models from synthetic source domains to real-world target distributions is a well-recognized bottleneck, particularly in comparable fine-grained visual classification tasks~\cite{smith2021leveraging}.

The reference images are characterized by uniform lighting, precise orthogonal alignment, the absence of lens distortion and flawless surfaces. In contrast, the target domain consists of physical photographs taken using industrial sensors or consumer webcams. These images exhibit a high degree of variability, including significant lens distortion, uneven ambient lighting, sharp shadows, broad specular reflections induced by protective polymer card sleeves, reflections from holographic foils, physical wear and tear and cluttered backgrounds (e.g., textured tables or playmats).

If a model is optimized purely on the source domain, its learned feature representations will be overly focused on artifact-free edges and uniform color palettes. As a result, the network becomes fragile and struggles to generalize to the reality of the target domain.

\section{Synthetic Data Generation}\label{sec:synthetic_gen}
To bridge the \hyperref[sec:domain_shift]{domain shift} described above, the training pipeline incorporates an on-the-fly synthesis module designed to simulate the artifacts inherent to physical image acquisition.

\subsection{Removing Sharp Edges}
Digital templates are often stored as rectangular images with transparent edges encoded via an alpha channel. To prevent the neural network from overfitting to these artificial, artifact-free edges, the cards are placed onto a stochastically generated background canvas, denoted as $\mathbf{X}_{\text{cv}}$. 

To construct this canvas, a base color is first sampled uniformly for each RGB channel, $c \sim \mathcal{U}(20, 230)$. To simulate sensor grain and the physical texture of a playing mat, additive Gaussian noise sampled from $\mathcal{N}(\mu=0, \sigma=4)$ is applied to every pixel and the resulting tensor is clipped to the valid 8-bit image range $[0, 255]$.

Once the canvas is generated, the card's alpha channel $\bm{\alpha}$ is explicitly extracted and linear interpolation is performed between the card's RGB tensor $\mathbf{X}_{\text{card}}$ and the noisy canvas $\mathbf{X}_{\text{cv}}$:
\begin{equation}
    \mathbf{X}_{\text{composite}} = \mathbf{X}_{\text{card}} \odot \bm{\alpha} + \mathbf{X}_{\text{cv}} \odot \left(\mathbf{1} - \bm{\alpha}\right)
\end{equation}
where $\odot$ denotes the component-wise\footnote{also known as the Hadamard product} product. This ensures that the synthetic images mimic the continuous edge blending observed when a physical card is placed on a textured surface, forcing the network to focus on the internal visual features rather than the bounding box boundaries. The complete generation and blending process is demonstrated in \reflst{lst:alpha_blend}.

\begin{lstlisting}[language=Python, caption={Stochastic background generation and alpha compositing.}, label={lst:alpha_blend}]
# generate stochastic noisy background canvas
base_color = np.random.randint(20, 230, (1, 1, 3), dtype=np.uint8)
canvas = np.ones((target_size, target_size, 3), dtype=np.uint8) * base_color
noise = np.random.normal(0, 4, (target_size, target_size, 3))
canvas = np.clip(canvas + noise, 0, 255).astype(np.uint8)

# resize template and calculate offsets
scale = (target_size * 0.9) / max(h, w)
new_w, new_h = int(w * scale), int(h * scale)
resized_card = cv2.resize(image, (new_w, new_h), interpolation=cv2.INTER_AREA)
x_offset = (target_size - new_w) // 2
y_offset = (target_size - new_h) // 2

# extract RGB, normalized Alpha, and Canvas ROI
bgr = resized_card[:, :, :3]
alpha_3d = np.expand_dims(resized_card[:, :, 3] / 255.0, axis=2)
roi = canvas[y_offset : y_offset + new_h, x_offset : x_offset + new_w]

# perform linear interpolation (Alpha Blending)
canvas[y_offset : y_offset + new_h, x_offset : x_offset + new_w] = \
    (bgr * alpha_3d) + (roi * (1.0 - alpha_3d))
\end{lstlisting}

\subsection{Augmentation Pipeline}
During every training iteration, every reference image is processed through a stochastic augmentation chain. To ensure the reproducibility of the trained models, the exact configuration of the implemented pipeline is detailed in Table~\ref{tab:augmentations}. The applied transformations are categorized by the physical phenomena they are designed to simulate:
\begin{itemize}
    \item \bb{Geometric Deformations}: Real-world physical cards are rarely photographed perfectly orthogonal to the camera lens. Transformations such as affine scaling, translation, rotation, and perspective warping are applied to induce model invariance to camera pose, alignment errors, and variations in focal length.
    \item \bb{Optical and Environmental Simulation}: Playing cards are frequently protected by glossy plastic sleeves, which introduce severe specular highlights and glare. To model this, operations such as \texttt{RandomSunFlare} and \texttt{GlassBlur} are utilized alongside standard motion blur and shadow generation. Furthermore, variations in brightness, contrast, and hue approximate diverse environmental lighting conditions (e.g., warm incandescent room lighting versus cool daylight).
    \item \bb{Sensor Noise and Occlusions}: Consumer webcams and smartphone cameras frequently introduce compression artifacts and ISO noise under low-light conditions. These are modeled using \texttt{ImageCompression} and \texttt{ISONoise}. Finally, physical damage is simulated using \texttt{CoarseDropout} with random noise filling acting as a spatial regularizer that forces the network to learn distributed feature representations rather than relying on a single localized focal point.
\end{itemize}

By applying this pipeline, the model is trained on a continuously varying stream of unique samples, bridging the domain shift mentioned in \refsec{sec:domain_shift} between digital templates and physical photographs.

It is important to note that while synthetic augmentation produces a highly robust baseline representation, completely eliminating the simulation-to-reality gap typically requires a final fine-tuning stage on a curated dataset of real-world images. However, manually acquiring such a dataset at the massive scale required for \ac{TCG} classification falls beyond the scope of this thesis. Real-world fine-tuning remains a primary avenue for future work; one that can be directly facilitated by deploying our proposed robot to automate the capture and pseudo-labeling of real-world photographic data.

\section{Theoretical Framework}

\subsection{Metric Learning Approach}
Standard image classification models typically employ Softmax classification with Cross-Entropy loss to predict a probability distribution over a fixed set of classes. While effective for closed-set problems (e.g., ImageNet classification), this approach is fundamentally unsuitable for \ac{TCG} identification due to
\begin{itemize}
    \item \bb{High Cardinality and Open-Set Nature}: \ac{MTG} contains over 100,000 unique card faces. While training a final fully connected layer of size $N \approx 10^5$ is computationally possible, deploying it as a fixed-size classifier is highly impractical. The domain is inherently open-set; new expansions are released frequently, rendering a fixed-size classifcation layer obsolete without constant retraining.
    \item \bb{Intra-Class Variance vs. Inter-Class Similarity}: The model must be invariant to non-semantic visual noise (e.g., varying lighting, card wear or foiling effects). Furthermore, the \ac{TCG} domain features extreme inter-class visual similarity, where reprinted cards often share identical artwork and differ only in tiny details like set symbols or edition text (see \reffig{fig:tiny_details} for example). While differentiating these tiny details can be delegated to a secondary verification pipeline, the primary model must still robustly group visually similar cards. Standard Softmax classification with Cross-Entropy loss maximizes inter-class separability but lacks explicit constraints to enforce intra-class compactness.
\end{itemize}
To address these challenges, a \ac{DML} approach was adopted. Instead of classifying images directly, the goal is to learn a mapping function $f: \mathcal{I} \rightarrow \mathbb{R}^d$, where $\mathcal{I}$ represents the input space of all possible card images, and $\mathbb{R}^d$ is a low-dimensional Euclidean feature space (typically constrained to the surface of a hypersphere). In this space, the geometric distance between embeddings corresponds directly to semantic visual similarity, allowing for card identification via Nearest Neighbor Search rather than direct classification.

\subsection{ArcFace: Additive Angular Margin Loss}\label{sec:arcface}
ArcFace (Additive Angular Margin Loss)~\cite{Deng_2022} was employed, a state-of-the-art objective function originally designed for large-scale facial recognition. The hypothesis is that \ac{TCG} recognition shares key characteristics with facial recognition: it requires subtle distinctions between thousands of classes based on fine-grained visual cues.

\subsection{Geometric Interpretation}
To understand ArcFace mechanics, it is best to start by analyzing the standard operations of the final fully connected layer of the classification network.

Let $\mathbf{I}_i \in \mathcal{I}$ denote the $i$-th input image in the current batch. The backbone network acts as the mapping function, extracting the deep feature vector (embedding) such that $\mathbf{x}_i = f(\mathbf{I}_i) \in \mathbb{R}^d$, where $d$ is the embedding dimension ($d=512$ in our architecture). 
Furthermore, let $\mathbf{W} \in \mathbb{R}^{d \times C}$ be the weight matrix of the final layer, where $C$ is the total number of classes (unique cards in our case).

The matrix $\mathbf{W}$ can be decomposed into $C$ column vectors $\mathbf{W} = \begin{bmatrix}\mathbf{W}_1, \mathbf{W}_2, \dots, \mathbf{W}_C\end{bmatrix}$, where each vector $\mathbf{W}_j \in \mathbb{R}^d$ acts as the learned \textit{class center} for the $j$-th class. In a standard linear layer (with bias set to zero), the activation (logit) for the $j$-th class given the embedding $\mathbf{x}_i$ is
\begin{equation}
    \text{logit}_j = \mathbf{W}_j^\intercal \mathbf{x}_i.
\end{equation}

Geometrically, this dot product relates to the angle $\theta_{j,i}$ between the feature vector and the class center:
\begin{equation}
   \mathbf{W}_j^\intercal \mathbf{x}_i = \|\mathbf{W}_j\| \|\mathbf{x}_i\| \cos \theta_{j,i}.
\end{equation}
In traditional Softmax classification with Cross-Entropy loss, the magnitudes $\|\mathbf{W}_j\|$ and $\|\mathbf{x}_i\|$ influence the output probability, which does not necessarily contribute to class separability. ArcFace addresses this by strictly enforcing normalization. By fixing $\|\mathbf{W}_j\| = 1$ and $\|\mathbf{x}_i\| = 1$ via $L_2$ normalization, the prediction depends \textit{solely} on the angle $\theta_{j,i}$:
\begin{equation}
    \mathbf{W}_j^\intercal \mathbf{x}_i = \cos \theta_{j,i}.
\end{equation}
This projects the learned features onto a hypersphere, where minimizing the loss equates to minimizing the geodesic distance (angle) between the sample $\mathbf{x}_i$ and its corresponding ground-truth class center $\mathbf{W}_{y_i}$.

To enforce a margin between classes, ArcFace adds an angular penalty $m$ directly to the angle $\theta_{y_i, i}$ of the ground-truth class. Additionally, since the cosine range $[-1, 1]$ is insufficient for the Softmax activation function to produce a sharp probability distribution (leading to slow convergence), the logits are scaled by a fixed factor $s$.
\input{./fig/tikz/arcface.tex}
The final ArcFace loss for a sample $\mathbf{x}_i$ with ground-truth label $y_i$ is defined as:
\begin{equation}
    L_{\text{ArcFace}} = -\log \frac{e^{s \cdot \cos(\theta_{y_i, i} + m)}}{e^{s \cdot \cos(\theta_{y_i, i} + m)} + \sum_{j \neq y_i} e^{s \cdot \cos(\theta_{j, i})}}
\end{equation}
By minimizing this loss, the network learns to map images of the same card to tightly clustered points on the hypersphere, while pushing different cards apart. Crucially, during the inference phase, the weight matrix $\mathbf{W}$ and the margin $m$ are discarded. The recognition is instead framed as a retrieval task: the system computes the cosine similarity between the extracted query embedding $\mathbf{x}_q = f(\mathbf{I}_q)$ and all pre-computed reference embeddings in the database. The final identity of the card is then determined via an $\arg\max$ operation over these cosine similarities, effectively selecting the nearest neighbor in the learned feature space.

\section{Model Architecture}
To ensure the network focuses exclusively on card features rather than background clutter, the complete system implements a two-stage pipeline, as illustrated in Figure~\ref{fig:pipeline_flow}. Phase 0 (Localization) utilizes an OpenCV-based detection script to locate the bounding box of the physical card within the raw camera frame, extracting it as a tightly cropped query image ($\mathbf{I}_q$). This image is then passed into the core neural network for feature extraction (Phase 1) and subsequent similarity matching (Phase 2).

The core neural network (\texttt{MTGReconModel}) utilized for the feature extraction is trained using a metric learning paradigm. Its underlying architecture is composed of three distinct modules designed to map raw pixel data into discriminative 512-dimensional Euclidean space.

\subsection{Backbone Network}\label{sub:backbone}
The foundational feature extractor is a \ac{ResNet}-50~\cite{He2015} architecture, instantiated via the \texttt{timm} library. To adapt the network for metric learning, the final classification fully connected layer is omitted. The backbone is initialized with weights pre-trained on the ImageNet dataset to leverage learned low-level visual features such as edge detection and basic geometric pattern recognition. The network processes standardized input tensors of size $512 \times 512 \times 3$. To prevent catastrophic forgetting of the pre-trained weights while adapting to the \ac{TCG} domain, the backbone parameters are optimized using a learning rate ten times smaller than that of the newly initialized custom network heads.

\subsection{Embedding Head}\label{sub:embedhead}
The 2048-dimensional feature maps extracted by the \ac{ResNet}-50 backbone are projected into a compact embedding space via a custom \ii{Embedding Head}. Acting as a dimensionality reduction bottleneck\footnote{In deep learning terminology, a \enquote{bottleneck} does not imply a computational performance constraint or throughput limitation, as it commonly does. Rather, it denotes a structural narrowing of the network. By forcing high-dimensional data through a lower-dimensional layer, the network is forced to discard redundant noise and compress the information into a dense, highly salient feature vector.}, this module consists of the following sequential application of common operations in deep learning:
\begin{enumerate}
    \item \ii{1D Batch Normalization}~\cite{ioffe2015batch}: Stabilizes the high-dimensional output from the backbone. During distributed training across multiple GPUs, this is converted to Synchronized Batch Normalization to ensure accurate batch statistic computation.
    \item \ii{Dropout}~\cite{dropout}: A dropout layer with a probability of $p=0.4$ is applied to regularize the network and mitigate overfitting, a common failure mode in fine-grained open-set classification.
    \item \ii{Linear Projection}: A fully connected layer without bias parameters reduces the dimensionality from 2048 to the target embedding size of $d=512$.
    \item \ii{1D Batch Normalization}: A final normalization step applied immediately before the features are projected onto the hypersphere.
\end{enumerate}

\subsection{ArcFace Head}\label{sub:arcface}
During the training phase, the 512-dimensional embeddings are passed to the \texttt{ArcFaceHead} module, which implements the additive angular margin penalty described in \refsec{sec:arcface}. The head maintains a learnable weight matrix initialized via Xavier uniform initialization.

The software implementation of this module adapts the official PyTorch implementation\footnote{\url{https://github.com/deepinsight/insightface/blob/master/recognition/arcface_torch/losses.py}} provided by the ArcFace~\cite{Deng_2022} authors. To optimize memory and computational overhead, the forward pass computes the angular margin selectively. The flow proceeds as follows:
\begin{enumerate}
    \item \ii{Cosine Similarity:} The raw logits are calculated as the dot product between the $L_2$-normalized embeddings and the normalized weight matrix, yielding a similarity matrix where each element is $\cos(\theta_{j, i})$.
    \item \ii{Margin Application:} Utilizing PyTorch's in-place operations within a detached gradient context (\texttt{torch.no\_grad()}), the specific similarities corresponding to the ground-truth classes ($\cos(\theta_{y_i, i})$) are isolated. These target logits are transformed back into the angular domain via the inverse cosine function ($\theta = \arccos(\cos\theta)$), shifted by the additive margin penalty ($\theta + m$), and projected back into the cosine domain as $\cos(\theta_{y_i, i} + m)$.
    \item \ii{Feature Scaling:} The updated similarity matrix is multiplied by the hyperparameter $s$ before being passed to the Cross-Entropy loss function.
\end{enumerate}
Because both the embeddings and the class weights are $L_2$-normalized, their dot products are bounded strictly to the interval $[-1, 1]$. As mathematically demonstrated by Wang et al.~\cite{wang2017normface}, passing values of such small magnitude directly into a Softmax activation produces an overly uniform probability distribution. This imposes a lower bound on the Cross-Entropy loss, which prevents gradient propagation and prevents the network from converging. Multiplying the modified logits by $s$ (set to $s=64.0$ in our implementation) defines the radius of the embedding hypersphere. This expands the logit range to $[-s, s]$, ensuring the Softmax function can generate the steep, confident probability distributions required for effective Cross-Entropy optimization.

As visualized in \reffig{fig:pipeline_flow}, it is important to note that this module is utilized exclusively during training. Upon deployment for inference, the \texttt{ArcFaceHead} weights are completely removed from the model's state dictionary. The system then performs recognition by routing the raw 512-dimensional vectors produced by the Embedding Head directly into the Similarity Matching engine (Phase 2).

\subsection{Dimensional Collapse}
During early training iterations, it was observed that the network would occasionally degenerate, producing an output vector approaching zero for all input images. In this state, the network effectively learns to ignore the input data, mapping all samples to a single point in the embedding space to trivially satisfy the intra-class distance minimization without learning meaningful inter-class separability.

To prevent this catastrophic failure and stabilize the training dynamics, several structural and optimization constraints were integrated into the pipeline:
\begin{enumerate}
    \item \ii{Differential Learning Rates}: The \ac{ResNet}-50 backbone is initialized with ImageNet pre-trained weights, while the embedding and ArcFace heads are randomly initialized. Using a uniform learning rate across the entire network risks destroying the pre-trained low-level feature detectors. Therefore, the backbone parameters are optimized with a learning rate an order of magnitude smaller ($\eta = 10^{-5}$) than the network heads ($\eta = 10^{-4}$).
    \item \ii{Strict $L_2$ Normalization}: By enforcing $L_2$ normalization on both the feature embeddings $\mathbf{x}_i$ and the class weights $\mathbf{W}_j$ prior to calculating the cosine similarity, the magnitudes of the vectors are factored out of the loss calculation. This mathematically prevents the network from minimizing the loss by shrinking the embedding magnitudes to zero.
\end{enumerate}
